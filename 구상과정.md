# 구상 과정

1. 6개 앙상블 vs sum(앙상블(2), 앙상블(2), 앙상블(2))
    - 6개 앙상블
        - 단일 검색 연산
        - 계산 복잡도, 메모리 사용량, 연산 시간 적음
        - 빠른 검색 가능
    - 2개 앙상블 후 merge
        - 계산 복잡도, 메모리 사용량, 연산 시간 많음
        - 높은 정확도 보장
2. Compression 도구 탐색
    1. LLMChainExtractor(LLMCF)
        - 자연어를 이해하는데(맥락 파악) 적합할 것으로 추정
        - LLM 사용해 관련성 높은 문서 추출
        - 쿼리 기반 스마트 필터링
    - CrossEncoderReranker(CER)
        - 문서 압축 도구로, 검색된 문서를 reranking하는 도구
        - Hugging Face의 [Cross Encoder 모델](https://huggingface.co/BAAI/bge-reranker-v2-m3) 사용
        - 초기 검색된 문서의 관련성을 더 정확하게 평가
    - EmbeddingFilter
        - 임베딩 유사도 기반 문서 필터링
        - 의미론적 유사성 평가

1. 구상 시나리오
    1. 6개 앙상블 + CER
        1. 높은 정확도와 관련성 → 검색 기능 최적화
        2. 정밀도가 높아서 LLM 기반 답변 생성 등 부담 감소
    2. 6개 앙상블 + LLMCF
        1. LLM 기반 컨텍스트 이해 → 높은 비용
        2. 복잡한 쿼리 대응 가능
        3. 일관성 및 비용 측면에서 비효율적
    3. 6개 merge + CER
        1. 검색 결과 1개의 일관된 리스트로 조정되므로 노이즈 감소
        2. merge 과정에서 일부 관련있는 결과 누락 가능성 있음
    4. 6개 merge + LLMCF
        1. 원문 형태 변형 가능성 있음
        2. merge해서 검색 속도 빠름
    5. 2개 앙상블 + merge + CER
        1. 재현율과 정밀도 균형 유지 가능
        2. 중독 문서, 최적화 기능 → 일부 중요 문서 누락 가능성
    6. 2개 앙상블 + merge + LLMCF
        1. LLM 응답 품질에 일관성이 없음
        2. 재현율과 정밀도 균형
        3. 중요한 세부 정보 사라질 수도 있음
    - vscode retriever까지만 실행
        
        
        | 경우 | vscode 실행 속도 | langsmith 속도 |
        | --- | --- | --- |

        | 개별 리트리버(6개) + LLMChainExtractor | 6.18 | 37.69 |
        | MergeRetriever(6개) + CrossEncoderReranker | 6.27 | 38.83 |
        | MergeRetriever(6개) + LLMChainExtractor | 6.13 | 36.04 |
        | (앙상블 후 Merge) + CrossEncoderReranker | 7.56 | 37.57 |
        | (앙상블 후 Merge) + LLMChainExtractor | 5.43 | 38.82 |
